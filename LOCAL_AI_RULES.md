# LOCAL_AI_RULES.md

Назначение: локальные инженерные правила для ИИ-ассистентов, работающих с репозиторием Sonarium.

Область действия:
- файл обязателен к чтению перед изменением кода;
- правила применяются ко всему репозиторию;
- файл не должен попадать в runtime-образ приложения.

## 1. Дисциплина проекта

- Перед изменениями читать `docs/architecture.md`, `docs/modules.md` и этот файл.
- Держать изменения узкими и привязанными к пользовательской задаче.
- При изменении поведения обновлять документацию в том же change set.
- Не вводить временные хаки без явной причины и понятного пути удаления.

## 2. Архитектурные ограничения

- Sonarium остаётся modular monolith.
- Бизнес-логика не должна жить в HTTP-обработчиках.
- Сервисы не зависят от transport-слоя.
- Работа с БД идёт через repository layer.
- Нельзя создавать циклические зависимости между модулями.
- Кросс-модульные вызовы идут через сервисы и интерфейсы, а не через прямой доступ к storage.

## 3. Docker и деплой

- Приложение обязано запускаться через Docker и Docker Compose.
- Изменения в runtime-конфигурации должны сопровождаться обновлением:
  - `docker-compose.yml`
  - `docker-compose.local.yml` при необходимости
  - `.env.example`
- Portainer deployment не должен ломаться из-за локальных dev-допущений.
- Volumes, healthchecks и startup path должны оставаться рабочими.

## 4. База данных и миграции

- Все изменения схемы оформляются отдельными versioned migration-файлами.
- Миграции должны быть детерминированными и forward-only.
- Нельзя скрыто менять схему вне migration layer.

## 5. UI и i18n

- Все новые пользовательские строки обязаны идти через i18n.
- Для новых ключей должны быть значения как минимум в `en.json` и `ru.extra.json`/`ru.json`.
- Нельзя оставлять английские строки в русском интерфейсе по умолчанию.
- Все текстовые файлы UI должны быть в UTF-8.
- Изменения в i18n сопровождаются тестами или обновлением существующих i18n-тестов.

## 6. Аудио и библиотека

- Изменения плеера не должны ломать:
  - восстановление состояния,
  - очередь,
  - прогресс-бар,
  - hover-сценарии,
  - загрузку и сканирование библиотеки.
- Массовая загрузка файлов должна оставаться наблюдаемой для пользователя.
- Любой background upload должен иметь явный прогресс и возможность отмены или скрытия.

## 7. Код и размер изменений

- Предпочитать небольшие функции с одной ответственностью.
- Крупные файлы разбивать по ответственности, если это не увеличивает хаос.
- Не складывать unrelated-логику в shared helpers без необходимости.

## 8. Тестирование

- Для новых критичных пользовательских сценариев добавлять тесты или усиливать текущие.
- Для i18n, конфигурации и migration-изменений проверка обязательна.
- После значимых backend-правок проверять `go build ./cmd/server`.
- После значимых frontend-правок проверять `node --check` для изменённых файлов.

## 9. Документация

- При изменении архитектуры, деплоя, env или release-поведения нужно обновлять:
  - `README.md`
  - релевантные файлы в `docs/`
  - `CHANGELOG.md` при пользовательски заметных изменениях

## 10. Запрещено

- Прятать бизнес-логику в transport handlers
- Ходить в БД в обход repository layer
- Добавлять runtime-зависимости от внешних CDN
- Коммитить локальные секреты, `.env`, временные бинарники и служебные артефакты
- Оставлять в проекте битую кодировку и mojibake-строки
